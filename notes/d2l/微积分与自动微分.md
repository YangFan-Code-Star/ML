# 2.4-2.5 微积分与自动微分

## 一句话总结

**大白话**：梯度就是「往哪个方向拧旋钮，损失下降最快」的指南针。以前要手算这个方向，现在 PyTorch 帮你自动算——你只管写前向公式，它记下每一步，调一句 `loss.backward()` 就把所有参数的梯度算好了。这是训练一切模型的发动机。

**严谨说法**：导数/[[梯度]]衡量损失对参数的敏感方向；PyTorch 的 `autograd` 通过构建计算图自动反向求导，把「手算梯度」变成 `loss.backward()`，是训练一切模型的引擎。

## 本节解决的问题

- 导数、偏导、梯度、链式法则在优化中起什么作用？
- 如何用 `requires_grad` 与 `backward()` 自动求梯度？
- 为什么要 `zero_grad()`？如何分离计算图 (`detach`)？

## 核心概念 / 公式 / API

### 梯度与链式法则
> **大白话**：梯度指向「上坡最陡」的方向，所以我们朝它反方向走就是下坡（减小损失）。链式法则就是「层层套娃求导」——外层对中层求导、中层对内层求导，乘起来；反向传播只是把这个过程自动化。

- [[梯度]] $\nabla_{\mathbf{x}} f$ 指向函数增长最快的方向，优化沿其反方向。
- 链式法则：$\frac{\partial y}{\partial x} = \frac{\partial y}{\partial u}\frac{\partial u}{\partial x}$，反向传播即其自动化。

### autograd 工作流
```python
x = torch.arange(4.0, requires_grad=True)  # 开启追踪
y = 2 * torch.dot(x, x)                     # 构建计算图
y.backward()                                # 反向求导
print(x.grad)                               # = 4x
x.grad.zero_()                              # 梯度累加，需手动清零
```

| API | 作用 |
| --- | --- |
| `requires_grad=True` | 标记需要梯度的叶子张量 |
| `y.backward()` | 从标量 `y` 反传，填充 `.grad` |
| `x.grad.zero_()` | 清零梯度（默认累加） |
| `x.detach()` | 从计算图分离，得到不带梯度的常量 |
| `with torch.no_grad():` | 推理时关闭追踪，省内存提速 |

### 非标量反向
- `backward` 默认要求标量；对向量需传 `gradient=` 或先 `.sum()`。

## 输入输出形状

- `x.grad` 与 `x` 形状一致。

## 最小可运行代码

仓库脚本：`d2l/code/2.4微积分.py`、`d2l/code/2.5自动微分.py`

```python
import torch

x = torch.arange(4.0, requires_grad=True)
y = 2 * torch.dot(x, x)
y.backward()
assert torch.equal(x.grad, 4 * x)        # 梯度 = 4x

x.grad.zero_()
u = (x * x).detach()                      # 把 u 当常量
(u * x).sum().backward()
assert torch.equal(x.grad, u)             # 梯度 = u（不回传到 x*x）
```

## 易错点

- 忘记 `zero_grad()` → 梯度跨 batch 累加，训练异常。
- 对非标量直接 `backward()` 报错——先 `.sum()` 或传 `gradient`。
- 推理/评估忘记 `torch.no_grad()`，浪费显存且变慢。

## 复习卡片

- Q: 为什么每个 batch 后要 `zero_grad`？
  A: PyTorch 梯度默认累加，不清零会叠加上一轮。
- Q: `detach()` 的作用？
  A: 把张量从计算图分离当作常量，阻断梯度回传。
- Q: 推理时如何关闭梯度？
  A: `with torch.no_grad():`。
