# 12.1-12.2 编译器与解释器、异步计算

## 一句话总结

**大白话**：「一行一行边写边跑」（命令式）好调试但有点慢；「先把整张计算图画好再统一优化」（编译，如 `torch.compile`）更快、好部署但难调。另外 GPU 是「下了单就走人」的异步模式——CPU 把任务丢给 GPU 就立刻返回了，所以想准确计时必须用 `synchronize` 等 GPU 真正算完。

**严谨说法**：命令式编程灵活易调试但有 Python 开销；符号式/编译（如 `torch.jit`/`torch.compile`）能优化和加速；PyTorch 的 GPU 运算是异步的，CPU 下发指令后立即返回，需 `synchronize` 才能准确计时。

## 本节解决的问题

- 命令式 vs 符号式编程的取舍？
- 如何把模型编译以提速/部署？
- 为什么 GPU 计时要显式同步？

## 核心概念 / 公式 / API

| 范式 | 特点 |
| --- | --- |
| 命令式 (eager) | 逐行执行，易调试，有 Python 开销 |
| 符号式 / 编译 | 先构图再优化，快、易部署，难调试 |

- `torch.jit.script` / `torch.compile`（新）把模型编译为优化图，提升吞吐、便于序列化部署。
- **异步执行**：GPU kernel 入队后 CPU 立即返回；`torch.cuda.synchronize()` 等待完成，计时/基准测试必须同步。

## 最小可运行代码

```python
import torch, time

x = torch.randn(4096, 4096, device="cuda")

torch.cuda.synchronize()
t0 = time.time()
for _ in range(10):
    y = x @ x
torch.cuda.synchronize()        # 等待 GPU 真正算完再计时
print((time.time() - t0) / 10)

# 编译加速（PyTorch 2.x）
compiled = torch.compile(net)
```

## 易错点

- 不 `synchronize` 直接计时 → 测到的是「下发时间」而非计算时间。
- 误以为 eager 模式慢就不能用——多数场景足够，编译用于部署/极致性能。
- `torch.compile` 首次运行有编译开销（预热后才快）。

## 复习卡片

- Q: 命令式与符号式各自优势？
  A: 命令式易调试，符号式/编译快且易部署。
- Q: 为什么 GPU 计时要同步？
  A: GPU 异步执行，需 `synchronize` 等待真正算完。
- Q: PyTorch 2.x 编译加速的 API？
  A: `torch.compile`。
